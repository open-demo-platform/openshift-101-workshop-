= Module 2: Deploying Applications

== Introduction

In this module, you'll learn how to deploy applications to OpenShift using multiple methods:

* Deploying from pre-built container images
* Building from source code using Source-to-Image (S2I)
* Scaling applications up and down
* Triggering rolling updates

**Estimated Time**: 60 minutes

[#container-image]
== Deploying from a Container Image

The simplest way to deploy an application is from an existing container image.

=== Using the Web Console

. Switch to the **Developer** perspective:
+
.. Click the perspective dropdown in the top-left
.. Select **Developer**
.. Ensure your project `workshop-{guid}` is selected

. Deploy the Parks Map application:
+
.. Click **+Add** in the left navigation
.. Click **Container images**
.. In **Image name from external registry**, enter:
+
[source,text]
----
quay.io/openshiftroadshow/parksmap:latest
----
.. Click the search icon (magnifying glass) to validate the image
.. In **Application**, select **Create Application**
.. In **Application Name**, enter: `parksmap-app`
.. In **Name**, enter: `parksmap`
.. Click **Create**

. Monitor the deployment:
+
.. You'll be redirected to the **Topology** view
.. Click on the `parksmap` deployment to see details
.. Watch the pod status change from `Pending` → `ContainerCreating` → `Running`

=== Verifying the Deployment

. Check that the pod is running:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get pods -n workshop-{guid}
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
NAME                        READY   STATUS    RESTARTS   AGE
parksmap-5d6f8c8b4f-xk2w7   1/1     Running   0          2m
----

. Get more details about the pod:
+
[source,bash,role=execute,subs="attributes+"]
----
oc describe pod -l deployment=parksmap
----

. View the application logs:
+
[source,bash,role=execute,subs="attributes+"]
----
oc logs -l deployment=parksmap
----

TIP: The `-l` flag uses label selectors. OpenShift automatically adds labels like `deployment=parksmap` to resources.

[#s2i]
== Building from Source with S2I

Source-to-Image (S2I) is an OpenShift feature that builds container images directly from source code without requiring a Dockerfile.

=== Understanding S2I

S2I works by:

1. Detecting the programming language
2. Selecting an appropriate builder image
3. Injecting your source code into the builder
4. Running the build process
5. Creating a new container image
6. Deploying the image

=== Deploying a Node.js Application

. Create a new application from source:
+
[source,bash,role=execute,subs="attributes+"]
----
oc new-app nodejs:18-ubi8~https://github.com/sclorg/nodejs-ex.git \
  --name=nodejs-app \
  -n workshop-{guid}
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
--> Found image abc123... (2 weeks old) in image stream "openshift/nodejs" under tag "18-ubi8" for "nodejs:18-ubi8"

    Node.js 18
    ----------
    Node.js 18 available as container is a base platform for building and running various Node.js 18 applications and frameworks.

    * The source repository appears to match: nodejs
    * A source build using source code from https://github.com/sclorg/nodejs-ex.git will be created
      * The resulting image will be pushed to image stream tag "nodejs-app:latest"
      * Use 'oc start-build nodejs-app' to trigger a new build

--> Creating resources ...
    imagestream.image.openshift.io "nodejs-app" created
    buildconfig.build.openshift.io "nodejs-app" created
    deployment.apps "nodejs-app" created
    service "nodejs-app" created
--> Success
----

. Watch the build process:
+
[source,bash,role=execute,subs="attributes+"]
----
oc logs -f bc/nodejs-app
----
+
TIP: Press `Ctrl+C` to stop following the logs once the build completes.

. Verify the build completed successfully:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get builds
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
NAME           TYPE     FROM          STATUS     STARTED         DURATION
nodejs-app-1   Source   Git@abc123    Complete   3 minutes ago   2m15s
----

. Check that the application deployed:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get pods -l deployment=nodejs-app
----

=== Understanding the Build Process

Let's explore what S2I created:

. View the BuildConfig:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get buildconfig nodejs-app -o yaml
----
+
This shows the build strategy, source repository, and output image.

. View the ImageStream:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get imagestream nodejs-app
----
+
ImageStreams are OpenShift-specific resources that track container image updates.

. View the Deployment:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get deployment nodejs-app
----

[#scaling]
== Scaling Applications

OpenShift makes it easy to scale applications horizontally (adding more pods).

=== Manual Scaling via CLI

. Check current replicas:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get deployment parksmap
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
NAME       READY   UP-TO-DATE   AVAILABLE   AGE
parksmap   1/1     1            1           10m
----

. Scale the `parksmap` deployment to 3 replicas:
+
[source,bash,role=execute,subs="attributes+"]
----
oc scale deployment/parksmap --replicas=3
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
deployment.apps/parksmap scaled
----

. Verify the scaling:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get pods -l deployment=parksmap
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
NAME                        READY   STATUS    RESTARTS   AGE
parksmap-5d6f8c8b4f-4l9xm   1/1     Running   0          15s
parksmap-5d6f8c8b4f-fqw8z   1/1     Running   0          15s
parksmap-5d6f8c8b4f-xk2w7   1/1     Running   0          10m
----

. Watch the pods scale in real-time:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get pods -l deployment=parksmap -w
----
+
TIP: Press `Ctrl+C` to stop watching.

=== Scaling via Web Console

. Go to the **Developer** perspective → **Topology**
. Click on the `parksmap` deployment
. In the **Details** tab, find the **Pod** section
. Click the up arrow to scale to 4 pods
. Click the down arrow to scale back to 2 pods

=== Scale Down

. Scale back to 1 replica:
+
[source,bash,role=execute,subs="attributes+"]
----
oc scale deployment/parksmap --replicas=1
----

. Verify:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get pods -l deployment=parksmap
----

== Triggering a New Build

S2I builds can be triggered manually to rebuild the application.

. Trigger a new build for the Node.js app:
+
[source,bash,role=execute,subs="attributes+"]
----
oc start-build nodejs-app
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
build.build.openshift.io/nodejs-app-2 started
----

. Watch the new build:
+
[source,bash,role=execute,subs="attributes+"]
----
oc logs -f build/nodejs-app-2
----

. List all builds:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get builds
----
+
You'll see both `nodejs-app-1` and `nodejs-app-2`.

== Rolling Updates

OpenShift automatically performs rolling updates when you change a deployment.

=== Updating an Image Tag

. View the current image:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get deployment parksmap -o jsonpath='{.spec.template.spec.containers[0].image}'
----

. Update the deployment to use a different tag (simulating an update):
+
[source,bash,role=execute,subs="attributes+"]
----
oc set image deployment/parksmap parksmap=quay.io/openshiftroadshow/parksmap:1.3.0
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
deployment.apps/parksmap image updated
----

. Watch the rolling update:
+
[source,bash,role=execute,subs="attributes+"]
----
oc rollout status deployment/parksmap
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
Waiting for deployment "parksmap" rollout to finish: 0 of 1 updated replicas are available...
deployment "parksmap" successfully rolled out
----

. View rollout history:
+
[source,bash,role=execute,subs="attributes+"]
----
oc rollout history deployment/parksmap
----

=== Rolling Back a Deployment

. Roll back to the previous version:
+
[source,bash,role=execute,subs="attributes+"]
----
oc rollout undo deployment/parksmap
----
+
.Expected Output
[source,textinfo,subs="attributes+"]
----
deployment.apps/parksmap rolled back
----

. Verify the rollback:
+
[source,bash,role=execute,subs="attributes+"]
----
oc rollout status deployment/parksmap
----

== Viewing Application Details

=== Using oc describe

. Get comprehensive details about a deployment:
+
[source,bash,role=execute,subs="attributes+"]
----
oc describe deployment parksmap
----
+
This shows replicas, selectors, pod templates, events, and conditions.

=== Viewing Events

. List recent events in your project:
+
[source,bash,role=execute,subs="attributes+"]
----
oc get events --sort-by='.lastTimestamp' | tail -20
----

Events show scaling, pod creation, image pulls, and errors.

=== Accessing Application Logs

. View logs from all pods with a specific label:
+
[source,bash,role=execute,subs="attributes+"]
----
oc logs -l deployment=parksmap --tail=50
----

. Stream logs in real-time:
+
[source,bash,role=execute,subs="attributes+"]
----
oc logs -f -l deployment=parksmap
----
